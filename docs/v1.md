type RegisterRequest struct {

  - 生成 access_token + refresh_token

  - 查找用户

  - 生成 token 对

  - 解析 refresh_token
  - 验证未过期
  - 生成新 token 对

每个 handler 需要：参数校验、调用 service、统一响应格式、Swagger 注解。

# 执行顺序建议

| 阶段 | Prompt | 内容 | 预计耗时 |
|------|--------|------|---------|
| **后端基础** | B0 → B1 | 项目初始化 + 用户认证 | 1-2天 |
| **后端核心** | B2 → B3 → B4 | 动作库 + 计划 + 训练记录 API | 2-3天 |
| **后端分析** | B5 | 数据分析聚合 API | 1天 |
| **前端 MVP** | F0 → F1 → F2 → F3 → F4 | 初始化 + 四大模块 | 5-7天 |
| **联调** | - | 前后端联调 + 同步测试 | 2-3天 |
| **后端扩展** | B6 | 模板 + PR + 挑战 | 1天 |
| **前端 V1.1** | F5 → F6 | PR + 肌群 + 云同步 | 3-4天 |
| **后端 V3** | B7 | 教练模式 | 1天 |
| **前端 V2-V3** | F7 → F8 | Watch + AI + 教练 + 视频 | 5-7天 |

> **核心原则**：后端先行，前端跟进。每个后端 Prompt 完成后用 curl / Postman 验证 API 正确性，再交给前端对接。
